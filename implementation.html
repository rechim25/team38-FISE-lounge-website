Implementation


Main Technologies

As previously mentioned in our Research section, our main technologies are React for the consumer application and Next.js & Node.js for our server API and administrator dashboard, making Javascript our primary project language.

Dependencies & Tools

Chakra UI

React library with built-in, stylized, responsive UI components. It is an actively maintained and popular library for React apps. Using Chakra UI has significantly improved our UI development speed and improved the quality of our Reacts components.

A-Frame

A library for creating interactive experiences within web applications through background animations. Allows us to generate the interactive VR background.

Jitsi

We needed an open-source, reliable and secure video-calling library that could be easily integrated into project. The answer was Jitsi Meet that allows our application to embed Jitsi video-call iFrames and provide our users with a simple way of contacting their friends and families.

Prettier

To maintain coding standards among group members we agreed to default to a code formatter. We chose the Prettier code formatter as it is one of the most opinionated Javascript formatter that allows custom syntax rules. TO be consistent in with our code base, we made use of the Prettier VScode plugin that allows us to specify workspace/project scoped formatter rules.

UI Design

UI Design Principles

Main objective of our UI design is to be suitable for elderly or people with less-technical expertise. For this reason, we have extensively researched UI alternatives and came up with a design that is based on gerontological principles.

First and foremost, we have created very accessible and suggestive command buttons that allow consumers to interact with our key functionalities directly.

The main idea behind putting the button in the corners is to eliminate visual clutters. On the sides the buttons are most visible.

To improve visibility, we have provided good contrast for buttons using a dark glassy panel with white colored icons. This will guarantee button visibility on all backgrounds.

The navigation depth of our UI is very small, allowing new less tech-savvy users to quickly grasp the means of interaction with our application. This is based on the gerontological research principle of avoiding nested & drop-down menus. Below you can see auto-generated 3d-modellings of our UI using Chrome features which prove that our navigation levels are very few and easy to access.

As general principles taken away from research papers, we abided by a no text policy, meaning we avoided it as much as possible on our main page. The reason behind this decision is that text poses questions to visibility and can often clutter the design. Also, dark blue and green have been proved to be detrimental to usability for elderly users and, thus, it is something we avoided as well on our main page.

UI Implementation Principles

UI has been implementation has been carried on with the intent of allowing further development and improvement in the future and thus have worked with the principles of code reusability. Reusability guarantees overall design consistency.

To illustrate this principle, let's look at how our command buttons in the main page are implemented. The CommandButton component is used to wrap all command buttons and provide consistency in design.

The component takes multiple parameters make it modular and reusable such as placement coordinates with respect to the main page (left, bottom, etc.) and box roundness properties. A custom function can be provided that is triggered when clicked. Very importantly is that we allow other components to be wrapped by it using the children attribute.

For instance, the "change background" button from our main page makes use of this component. by specifying the top=0 and left=0, meaning top-left corner. and setting the bottomRight corner to be rounded. It also, provides a custom function handleChangeScene that will change the currently selected background when the button is clicked.

In this usage case, we can see that an icon component is passed to the CommandButton. Thus, the icon will be rendered inside the button.

Voice Commands

Overview

A key feature of our application is to include 2 services for handling voice commands that can be chosen by the administrator from our dashboard. Below is how the dashboard option looks like for the administrator. When Cloud Features are enabled, IBM Watson services will be used. Otherwise, FISE Ask Bob will provide the virtual assistant service when the checkbox is disabled. When the administrator changes the cloud option of a consumer, a request is sent to our server to change the configuration of the respective consumer in the database.

The consumer configuration is cached in the browser's local storage as in the first figure below. When the cosnumer enters the application gets consumer at the /api/otc/:otc API endpoint by using the consumer's associated One-Time code (OTC). The response is transformed into JSON format and added the local cache with setItem() function.

Then, after the consumer configuration is fetched, we store it into a global variable that is availabe for use everywhere in the main component. In case no configuration is found, we redirect the consumer to the Onboarding page to login with the OTC.

Recording Voice Commands

Since the 2 afore mentioned services require different audio file types to process, we needed to integrate 2 libraries that can recording audio from browser as MP3 (for Watson) and Wav (for Ask Bob).

But using 2 libraries within the same component can increase code complexity dramatically. Thus, we have implemented a RecorderAdapter class that abstracts away the underlying differences of the 2 libraries using the Adapter design pattern. The class provides the necessary methods for recording audio such as start(), stop() and clean() the recording using asynchronous functions to provide flexibility when using the class in different ways. When a RecorderAdapter object is initialized at runtime it expects a isCloudEnabled boolean parameter which represents the user option for IBM Watson (if true) or Ask Bob (is false). This way the object will use the right recording library (MicRecorder for MP3 or Recorder for Wav as below).

RecorderComponent is a reusable component that allows integration of recording anywhere within the application. This component makes use of the RecorderAdapter class by calling its start, stop functions whenever and event is triggered. It also fetches the audio data to the selected APIs and handle the response data accordingly. In addition, it can render different things based on the state that it is in. For example, when a in recording state, we pass in a function that will render the microphone icon in red action feedback.

In the case of the Ask Bob service, RecorderComponent creates a new RecorderAdapter object with isCloudEnabled parameter set to true. When a user clicks on the component, the handleMicrophoneClick function in triggered. This will check if the component is already in a recording state and if not, then it will get access to the user's microphone with our initUserMedia() function and start the recording with the start() function of the adapter (as seen below).

Using the API for Voice Commands

Once the previously mentioned recording process is finalized comes the fetching phase of the voice command. The RecorderComponent supports integrating 2 different API services within it. As you can see below, it expects 2 pairs of functions made of an API fetch and a response handler that uses the data retrieved by the API in a custom way.

If the user option is set use IBM Watson isCloudEnabled === "true, the RecorderComponent will use the watsonFetch function to send the audio file (res) to our server which will then be redirected to IBM Watson for interpretation. Afterwards, watsonResponseHandler will use the response data to check for discovered user intents (i.e. "Call John", "change background") and the actions needed to carry out.

On the other hand, if the Ask Bob option is selected, then the user device will directly communicate with Ask Bob service by running the askbobFetch function (see above) without the need of our server as a mediator. This design decision improves response time and as in the case of running Ask Bob locally as there is no need for a server connection in this case.

Very important to note is the difference between the Watson and Ask Bob response handlers. As Ask Bob provides access to the whole FISE ecosystem of applications and API, the askbob response handler function needs to take care of more types of responses. For example, when Ask Bob detects an intent for reading recipes in the voice query of a user, it will request the FISE Concierge app to provide a recipe from their API and this response needs to be handled correctly on our side as well (see below how we handle Concierge's custom response and how IBM Watson response is handled).

As you can see the response handler forAsk Bob has a case for reading specific Concierge response such as recipe instructions which are separated by steps. The customResponse function is given to the RecorderComponent to be ran when any of the 2 services return a response. In our case, the customResponse is the function that plays Ask Bob’s or IBM Watson’s response out loud for the user to hear (to understand how speech synthesis is done, click here).

In conclusion, we allow the functionality of a virtual assistant to be reused in our application with the option of adding other pairs of services and hook functions.

Emergency Messages

Previously, we mentioned that the RecorderComponent is reusable. Here, we provide an example of how the design of the component has benefited the development process.

Emergency messages reuse the RecorderComponent for capturing short emergency audio recordings and then transcribe the audio into text to be sent as an alert to every contact of the consumer. It uses the RecorderComponent in almost the same way as voice recordings do, but with different response handlers (see below the Ask Bob response handler for emergency messages). The fetching functions are the same: audio file is sent to either Ask Bob or IBM Watson where speech is transformed into text. Now, the response handler does send the text response further to be read out to the user but instead sends it to our server's /api/otc/emergency API endpoint where the text is sent in batches on email and SMS to every contact.

Plugins

Plugins are rendered within a custom iFrame in the PluginComponent. The main objective is to allow other developers to extend the plugins component to their users’ needs and add any service they desire. For example, we have added games such as chess. Also, we integrated the BBC live news feed with the RSS protocol using vanilla JavaScript and no 3rd party libraries.

Developers can extend the frame in 3 steps:

  • First, initialize they’re data using JavaScript or 3rd party libraries.
  • Then, add their HTML in the return statement of the Plugin component .
  • If needed they can then add CSS code in the /styles/plugins.css file to style the Plugin component however they want.

Login Mechanism

Administrators

We use Passport.js a lightweight library for managing authentication. It is important for persisting login sessions on the server–side using the username-password strategy. For maintaining the session on the client-side we send an encrypted session token using the hapi/iron library. This token is sent to the server-side, which can check the validity of the token and allow access to the client-side without prompting the user for the credentials again.

Consumers (elderly)

Once we entered the application, the elderly consumer needs to enter the One-Time Code (OTC) that the administrator gave to enter the app. Once the OTC is entered, our app will send a query to the server to grab the consumer data at /api/otc/[otc]. Once this data is returned from the database it will be stored in the browser cache, not needing to be used again for improving accessibility. The One Time Code is designed to be memorable and easy to write in case it may need to be re-entered.

Other Privacy Features Highlights

Local Speech Synthesis

To provide additional privacy-safety features we allow users to benefit from the WebAPI Speech Synthesizer available in most browsers. This is an alternative to IBM Watson Speech-to-Text and can be switched on from the dashboard.

In the case of cloud enabled, the behavior of the playTextToSpeechLocally function is the following: receives text as paramter, then fetches the /api/otc/watson/text-to-speech endpoint on our server to synthesize the speech. The response will be an Mp3 blob that is turned into a blob and injected into an audio element in the root window.

In case the user starts a new recording while voice response is still playing, we want the audio to stop and in order to not overlap with what the user is saying. To achieve this we attach the stopSpeech function as a hook to the RecorderComponent, that stops the audio from playing when a new recording is initiated. It check if the audio is playing both in the case when IBM Watson or Ask Bob is enabled and, if it is, then it will stop the audio.

Running the Application Locally

As mentioned in our research Docker allows us to contain user data on the local device, improving privacy, while providing cross-platform capabilities. Using Docker Compose we have containerized all our services (App, Server,MongoDB, Ask Bob, Concierge) and allowed them to communicate on the private Docker host network.

On our docker branch there is an installation batch file that would install docker, therefore simplifying the local deployment process, enabling fast deployment in care-homes and other suitable places.
